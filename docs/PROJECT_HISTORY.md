# 历史决策与对话摘要

最后更新日期：2026-04-10

这份文档不是简单 changelog，而是帮助后续接手者理解“为什么会变成现在这样”。

## 1. 产品方向演进

### 初始需求

用户最初提出的是一套面向职业房东 / 二房东的租房管理微信小程序，包含：

- 房源管理
- 租客与合同管理
- 账单与财务引擎
- 水电抄表
- 智能催缴
- 首页看板

### 后续调整

经过多轮讨论，产品方向明确收敛为：

- 核心是房东自我记账与管理
- 不再把催缴文案作为重点
- 系统更强调：
  - 待收可视化
  - 记收
  - 凭证留存
  - 历史追溯

### 技术方向确认

用户明确确认：

- 技术栈：微信小程序技术栈
- 前端：Vue 3 + Uni-app + Tailwind CSS
- 后端规划：Node.js + Express/NestJS
- 数据库规划：MySQL + Prisma
- 存储：Cloudflare R2

## 2. 原型复刻过程中的关键决策

### 页面建设路线

用户要求：

- 新建一个项目
- 原型尽可能一致
- 逐页读取原型
- 每页做完后逐页校对

### 小程序优先

过程中明确确认：

- 目标是微信小程序，不是普通 Web 应用
- H5 只用来辅助调试

### tab 栏方案演进

曾尝试：

- 自绘悬浮 tabbar

后续因为：

- 小程序里看不到
- 切页白屏
- 出现超时和路由异常

最终回退为：

- 原生小程序 tabBar
- 通过 `switchTab` 提升稳定性和切换速度

## 3. 交互设计关键决策

### 房间详情与办理入住

用户多次强调：

- 不要做成普通全屏页面
- 要更像从下方弹出的抽屉页
- 当前情况和历史入住情况要分视图
- 历史入住情况按时间轴组织

因此当前实现形成了：

- 房间详情页
- 办理入住页
- 内部抽屉
- 时间轴组件

### 历史入住

这是一个重要强需求：

- 每个房间都要有历史入住情况
- 要支持当前租约和历史租约共存
- 退租后要进入历史

### 结构管理

用户对结构管理反复校正过：

- 结构管理要和工作台顶区保持一致
- 顶部区要固定
- 新建房间要是楼层最后一个虚框加号卡
- 新建楼栋也要遵循类似逻辑
- 删除入口要更精致，不要粗糙的文字按钮

## 4. 工程实现上的关键决策

### mock 数据与本地持久化

由于当前没有正式后端，已做出的折中方案是：

- `rentStore.js` 维护核心 mock 数据
- 使用本地存储做持久化
- 支持重置 demo 数据

这意味着：

- 页面间数据可以同步
- 刷新后状态不会丢
- 但还不是真正数据库落库

### 路由稳定性

过程中小程序曾出现：

- `routeDone with a webviewId ... is not found`
- `navigateBack with an unexist webviewId ...`
- `timeout`

因此增加了：

- `src/utils/navigation.js`
- 路由节流
- 防重复跳转
- 安全返回逻辑

### WXSS 兼容性

小程序编译过程中暴露出大量 Tailwind 兼容问题，已做过这些处理：

- 去掉不安全 Tailwind 选择器依赖
- 增加 `utilities.css`
- 增加构建后清理脚本
- 清理 `app.wxss` 中不兼容内容

## 5. 迭代中已经确认过的偏好

- 顶部区域应冻结，下部可滚动
- 小程序常见返回 / 关闭图标优先，不用文字按钮
- 不要堆很多重复统计
- 结构管理、账务、我的都要更像原型
- 账务页下半部分要更精致，按钮文案要正常
- 个人信息卡不能是白底

## 6. 当前阶段的真实状态

到 2026-04-10 为止：

- 页面结构已经基本齐全
- 主要业务闭环已经能演示
- 但是仍有不少 UI 精修和编码清理工作未完成

最突出的遗留问题：

- 页面乱码尚未完全清理
- `我的` 页和部分 mock 文案污染较多
- 后端尚未开始正式搭建

## 7. 后续开发时要延续的原则

1. 不要一上来重构整个项目
2. 先保住小程序可用性
3. 再继续贴近原型
4. 再接入正式后端
5. 每次改动都更新文档
