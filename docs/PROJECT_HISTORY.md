# 历史决策与对话摘要

最后更新日期：2026-04-10

这份文档用于帮助后续接手者理解项目为什么会演进成现在这样。

## 1. 产品方向演进

### 初始需求

用户最初希望做一套面向职业房东 / 二房东的租房管理微信小程序，覆盖：

- 房源管理
- 租客与合同管理
- 账单与财务引擎
- 水电抄表
- 智能催缴
- 统计报表

### 后续调整

经过多轮确认，产品方向已经收敛为：

- 核心是房东自我记账与管理
- 不再把催缴文案作为重点
- 系统强调待收可视化、记收、凭证留存和历史追溯

### 技术方向确认

用户已明确确认：

- 前端：Vue 3 + Uni-app + Tailwind CSS
- 目标端：微信小程序
- 后端规划：Node.js + Express/NestJS
- 数据库规划：MySQL + Prisma
- 文件存储规划：Cloudflare R2

## 2. 原型复刻过程中的关键决策

### 小程序优先

过程中已经明确：

- 目标是微信小程序，不是 PC Web
- H5 只用于辅助调试

### tab 栏方案演进

曾尝试使用自绘 tabbar，但后来因为看不见、切换慢、路由异常等问题，最终回退为：

- 原生小程序 tabBar
- `switchTab` 切页
- 稳定性优先于极致样式自由度

### 抽屉式交互

用户反复强调：

- 房间详情和办理入住应更像底部抽屉页
- 当前情况和历史入住要分视图
- 历史入住要按时间轴展示

因此项目里形成了：

- 房间详情页
- 办理入住页
- 内部抽屉组件
- 历史时间轴组件

### 结构管理的关键要求

用户多次校正过结构管理页：

- 顶部区要固定
- 新建房间应是楼层末尾的虚线加号卡
- 新建楼栋应遵循同类交互逻辑
- 删除入口要轻量，不要粗糙文字按钮

## 3. 工程实现上的关键决策

### mock 数据与本地持久化

由于当前没有正式后端，采用的是：

- `rentStore.js` 维护核心 mock 数据
- 本地存储做持久化
- 支持重置 demo 数据

这保证了：

- 页面间数据能同步
- 刷新后状态不丢
- 但还不是真正数据库落库

### 路由稳定性

项目迭代中出现过：

- `routeDone with a webviewId ... is not found`
- `navigateBack with an unexist webviewId ...`
- `timeout`

因此新增了：

- `src/utils/navigation.js`
- 路由节流
- 防重复跳转
- 安全返回逻辑

### WXSS 兼容性

微信小程序对部分 Tailwind 选择器不兼容，因此引入了：

- `src/styles/utilities.css`
- `scripts/sanitize-mp-weixin-wxss.cjs`
- `scripts/postprocess-mp-build.cjs`

当前仍需要继续在“高保真”和“小程序兼容”之间做平衡。

## 4. 重要视觉与交互偏好

已经明确的偏好包括：

- 上部固定，下部滚动
- 返回 / 关闭按钮采用主流小程序图标按钮
- 页面不要堆太多重复统计
- 工作台、账务、我的都尽量贴近原型
- 个人信息卡不能是白底
- 账务页更适合作为“流水历史页”而不是待收中心

## 5. 2026-04-10 这轮更新

本轮新增的明确决策：

- `账务` 页面不再展示待收视角，改为只看过往已收流水
- `账务` 页顶部总览改为累计实收口径
- `我的` 页面重做为更清晰的深色头卡 + 双分组菜单结构
- 文档维护已纳入默认流程，每次开发都应同步更新文档并提交 Git

## 6. 当前阶段真实状态

截至 2026-04-10：

- 页面结构已经比较完整
- 主要业务闭环已能演示
- 但仍有部分历史乱码和细节样式未清干净
- 后端正式开发尚未开始

## 7. 后续开发原则

1. 不要一上来就整体重构
2. 先保证小程序可用性
3. 再继续逼近原型
4. 再引入正式后端
5. 每次改动都更新文档并提交 Git
