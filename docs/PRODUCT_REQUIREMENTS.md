# 产品需求与原型约束

## 1. 产品定位

### 目标用户

- 拥有多套分散式房源的二房东
- 拥有集中式公寓的职业房东

### 核心价值

- 告别纸笔和 Excel
- 实现房源状态可视化
- 实现账单计算自动化
- 实现收款核销留痕
- 实现租客档案电子化

### 业务方向修正

项目最初有“催缴文案”设想，但已在后续迭代中调整为：

- 主要用于房东自我记账管理
- 不再强调自动生成催缴文案
- 不再把“复制催缴文案”作为核心能力

保留的能力是：

- 查看待收 / 欠费
- 手动记收
- 留存凭证

## 2. 业务闭环

### 闭环主线

1. 录入房源
2. 空置房办理入住
3. 生成分期账单
4. 水电抄表生成杂费
5. 记收房租 / 记收水电 / 记收灵活收费
6. 上传凭证（当前为模拟）
7. 办理退租
8. 归档历史入住记录

### 初始化流程

1. 配置全局水电单价模板
2. 新建房产院落
3. 批量添加楼栋与房间号

### 入住交房流程

1. 在首页选择空置房
2. 设定租期与租金规则
3. 填写租客身份明细
4. 强制上传身份证与合同扫描件
5. 系统计算首期与押金
6. 确认收款
7. 房间状态变为已租

### 日常收租流程

#### 固定周期租金

1. 系统根据租期和周期生成待收账单
2. 房东查看待收状态
3. 房东线下或微信收款
4. 房东在系统中点击记收
5. 可选上传凭证截图
6. 当期账单状态变为已收

#### 水电物业杂费

1. 房东现场抄表
2. 系统对比上期底数自动算费
3. 生成独立杂费单
4. 房东记收

### 退租流程

1. 录入最终水电读数
2. 计算应退押金或补交金额
3. 确认清退
4. 系统切断租约
5. 清空抄表状态
6. 房间恢复为空置
7. 上一任租约进入历史档案

## 3. 功能需求

### 房源管理

- 支持房产 / 楼栋 / 楼层 / 房间多层结构
- 支持状态展示：空置、已租、临期、欠费
- 支持结构管理：新增院落、楼栋、楼层、房间
- 房间资产照片上传与管理

### 租客与合同管理

- 录入租客姓名、电话、身份证
- 强制归档身份证和合同
- 全局档案库支持搜索
- 支持按姓名 / 房号 / 手机号模糊搜索

### 账单与财务

- 按支付周期生成分期账单
- 支持押金、首期、自定义金额
- 支持房租、水电、灵活收费三类流水
- 支持记收与凭证留痕

### 水电抄表

- 支持全局默认水电单价
- 房间可覆盖默认单价
- 录入底数与本期读数自动算费
- 可上传表计照片留档

### 历史留痕

- 所有影响财务或房态的操作进入历史记录
- 每个房间都要保留历史入住情况
- 历史入住情况按时间轴展示

### 报表

- 首页全局看板
- 账务页年度总览
- 未来支持导出 Excel

## 4. 非功能需求

### 易用性

- 高度适配移动端
- 单手操作优先
- 核心入口突出
- 抽屉式交互优先

### 安全性

- 未来需要 JWT 鉴权
- 身份证 / 合同 / 收据等敏感资料需要云端鉴权存储
- 对象存储方案预定为 Cloudflare R2

### 性能

- 小程序首页房态矩阵渲染应快
- 账单生成逻辑未来放后端异步

### 可靠性

- 财务与房态操作必须可追溯
- 退租、入住、记收、抄表都要留痕

## 5. 原型约束

### 视觉与交互

- 以 `C:\\Vibe\\rent.txt` 为主要原型参考
- 三个一级页面：工作台、账务、我的
- 房间详情和办理入住强调“抽屉式页面”
- 账务页和我的页顶部区域要求固定，下半区滚动
- tab 栏使用原生小程序 tabBar，优先切换稳定性

### 已确认的设计偏好

- 小程序优先，不做 PC 风格
- 结构管理支持编辑态
- 新建房间应采用楼层末尾虚线加号卡
- 新建楼栋应采用列表末尾新增卡
- 返回与关闭按钮使用主流小程序图标按钮，不用文字“返/关”

## 7. 当前阶段结论

- 当前产品阶段仍然是单用户 / 单端原型
- 后续可以演进到多租户
- 支付周期必须可配置，不能锁死
- 账务系统以“记账与留痕”为主，不再做催缴文案核心能力
